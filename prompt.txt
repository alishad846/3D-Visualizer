Feature: Bulk Product Import via CSV or Excel

Route
POST /api/products/bulk-upload/:projectId

Extract projectId from route param — never from the file
Verify ownership of projectId against current authenticated user before any file parsing begins
If ownership check fails, reject immediately with 403


File Acceptance

Accept only .csv and .xlsx
Multer memory storage
Reject if file size > 2MB
Reject if parsed row count > 100
Both checks before any validation logic runs


Required Columns in File
Re-check schema.sql and productController.js to confirm, but based on current understanding:

name — non-empty string, NOT NULL
category — non-empty string, NOT NULL

These two must be present in every row. If either is missing or empty, reject that row.

Optional Columns in File — Null Accepted
Carefully verify each of these against schema.sql for their actual DB nullability and defaults:

tagline
description
brand
sku
thumbnail_url — if present, validate it is a valid URL string pointing to our Supabase storage domain. No HTTP liveness check.
usdz_url — same URL format validation as above
price — must be numeric if present
currency — string, defaults to INR if absent
buy_url — valid URL format if present
qr_label
features — pipe-separated string in cell: Feature1|Feature2. Parse server-side into array before insert.
specs — pipe-separated key:value pairs in cell: Weight:200g|Height:10cm. Parse server-side into array of objects [{ key, value }] before insert.


Columns Excluded From File Entirely — Server Handles These

project_id — taken from route param
model_url — set to null on bulk insert, user attaches model later
usdz_url — wait, this is in optional above, do not duplicate
gallery_urls — excluded from bulk import entirely, do not accept in file
slug — server generated from brand + name, same logic as current single product flow: lowercase, replace non-alphanumeric with -, trim hyphens, fallback to product, append short random suffix if collision. Generate all slugs in memory for the batch, then resolve collisions in one single WHERE slug = ANY($1) query against DB.
status — hardcoded to 'draft' for all bulk inserted products
id, created_at, updated_at — DB generated


Two-Gate Validation — Strict Order
Gate 1 — File-level header validation:

Normalize all headers: trim whitespace, lowercase
Check all required columns are present
Check no unknown/unexpected columns exist
If gate 1 fails → reject entire file immediately, return which headers are missing and which are unexpected. Do not proceed to row parsing.

Gate 2 — Row-level validation:

Validate every row fully before touching the DB
Collect all valid rows and all error rows separately
Do not insert anything until all rows are evaluated
For each invalid row return:

json{
  "row": 4,
  "field": "name",
  "error": "Name is required",
  "solution": "Provide a non-empty product name in the name column"
}

Duplicate detection within the file itself: if two rows share the same name or sku, reject the second occurrence with a clear error
Duplicate detection against DB: check sku collisions within the same project in one single query for the whole batch, not per row


DB Insertion — Only After All Validation Passes

Insert valid rows only
Chunk size: 10 rows per chunk
One transaction per chunk covering both tables: products insert and qr_codes insert
For each inserted product, immediately create a qr_codes record in the same transaction:

product_id = newly inserted product id
project_id = from route param
user_id = current authenticated user
qr_token = generate unique token same way current single product flow does
destination_url = ${CLIENT_URL}/p/${slug}
qr_image_url = generate QR image same way current flow does, store to Supabase, save URL
scan_count = 0
is_active = true


If a chunk transaction fails, log it, skip that chunk, continue with next chunk. Do not abort the entire upload.


Rate Limiting

Apply a separate rate limit on this endpoint
Stricter than the normal product create endpoint
