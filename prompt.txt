Technical Architecture Audit: Mobile AR & WebXR Readiness
ScanVista Systems Engineering & Mobile AR Architecture Review

This audit provides a brutally honest, production-grade technical evaluation of the current state of mobile Augmented Reality (AR) readiness across the ScanVista platform. It identifies critical architectural bottlenecks, device-compatibility boundaries, database schema limitations, and physical scaling conflicts that must be resolved before releasing the system to creators and consumers.

1. Current System Summary
ScanVista's front-end consists of a modern React Three Fiber (R3F) 3D viewer wrapped around a PostgreSQL/Express backend. While the 3D rendering pipeline has been stabilized for standard viewports using custom container groups and adaptive cameras, the AR readiness is in an early sandbox stage.

The current implementation relies on a custom R3F WebXR integration (ARViewer.jsx) utilizing @react-three/xr for immersive-ar hit testing. While technically interesting, this raw WebXR path is highly fragile on iOS (which does not natively support WebXR in Safari) and introduces severe visual conflicts on Android.

2. Current Strengths
Centralized 3D Rendering: The consolidated ProductCanvas.jsx and non-mutating ProductModel.jsx establish a clean, predictable, and single-source-of-truth render tree for normal 3D presentation.
Database Hooks for Basic Metrics: The qr_scans table already possesses an ar_used column, and the user_interactions table has an 'ar_used' event state, laying the foundation for basic telemetry.
Basic WebXR Detection: The frontend correctly utilizes navigator.xr.isSessionSupported('immersive-ar') to prevent launching XR sessions on completely unsupported devices (e.g., standard desktop browsers).
3. Current Weaknesses
Broken iOS AR Support: iOS Safari does not support the WebXR Device API out of the box. The custom @react-three/xr canvas completely fails on Apple devices. There is no Apple AR Quick Look fallback implemented.
Viewer vs. AR Scale Destruction: The auto-scaling normalization pipeline forces all models (whether a 15-inch laptop or a 6-foot sofa) to fit exactly inside a 2.5 unit bounding box. While great for screen-framing, this destroys real-world scale. If pushed to AR, a sofa would appear as a tiny toy, and a wristwatch would appear giant, ruining spatial realism.
Lack of Asset Formats: The platform stores and serves .glb files only. iOS Quick Look requires Apple's native .usdz format. Without a converter pipeline or USDZ storage path, Apple users are locked out of native AR.
Heavy Custom AR Hit-Test Setup: The current ARViewer.jsx implements raw hit-testing math inside a custom WebXR frame loop. This is high-maintenance, fragile, and lacks standard UX polish (such as ground-plane reticles, scale gestures, and lighting estimation).
4. Hidden Production Risks
VRAM-Induced Crash Loop on iOS: Creators upload raw .glb files with uncompressed 4K or 8K textures. Standard mobile browsers (especially Safari) have tight VRAM ceilings. When Safari hits a VRAM limit during AR initialization, the iOS WebKit subsystem will silently crash and force-reload the page, creating a terrible user experience.
Silent Hit-Test Failures on Android: Because the system uses raw WebXR hit-testing, minor variations in Android device depth-sensing APIs can cause the placement reticle to drift, sink into the floor, or fail to resolve completely.
Z-Fighting and Missing Bottom Geometry: Many creator-uploaded models have open bottom meshes (no physical faces under the base). In AR, if the model is placed slightly below or exactly at the floor boundary, it will look hollow or flicker heavily due to the lack of ground-plane occlusion.
5. Hardcoded Assumptions Detected
autoRotate in ScannedProductUI.jsx: Hardcoded to true.
Universal targetSize = 2.5: Assumes every product should have the same bounding dimensions in all visual contexts.
AR WebXR Session Target: The capability layer assumes that if the device is a mobile phone, WebXR is the preferred path, neglecting the massive iOS Safari demographic.
LocalStorage Mockups: The ProductForm.jsx saves models directly to local blob URLs and mocks product creation in localStorage, concealing the lack of an optimized server-side processing pipe.
6. Mobile Risks
Thermal Throttling: Running a custom R3F/WebXR canvas on mid-range Android devices drives the GPU/CPU to 100% capacity. Within 2-3 minutes, thermal throttling triggers, dropping rendering frame rates to single digits and making AR placement sluggish.
Portrait/Landscape Reorientation: If a user rotates their device while placing an object in AR, the WebXR reference space can shift, displacing the model far from its original spot or locking the tracking system.
7. WebXR Risks
Browser Runtime Fragmentation: WebXR behaves differently on Samsung Internet, Chrome Mobile, Opera Mobile, and native in-app webviews (like Instagram or Facebook browsers). An AR session launched from inside a social media webview is highly likely to fail or crash due to strict iframe permissions.
8. iOS Ecosystem Risks
Lack of USDZ Delivery: Serving GLB files to Apple devices means the system must fall back to non-immersive 3D, entirely removing the "Real-World AR" value proposition for iPhone users (approx. 50% of the mobile market).
9. Android Ecosystem Risks
Google Play Services for AR (ARCore) Dependency: If a user's Android phone does not have ARCore pre-installed or updated, WebXR sessions fail silently or lock up on a black camera screen.
10. Rendering Architecture Risks
Monolithic Canvas Control: Although ProductCanvas successfully consolidated normal 3D rendering, wrapping WebXR inside the same canvas structure poses risks. WebXR takes complete control over the render loop, which can override custom camera positioners, aspect ratio calculations, and orbit controls.
11. Backend/Data Model Risks
Database Schema Gaps:
No usdz_url column to store iOS-compatible models.
No physical_dimensions (width, height, depth in meters) column. Without this, the system cannot verify or override real-world scales.
No poly_count or texture_resolution metadata to prevent high-end, uncompressed models from being served to mobile devices.
Storage Pipeline Constraints: Currently serves assets statically. It does not support on-the-fly model optimization, Draco compression, or asset scaling.
12. Future Scalability Risks
No Quality Level Tiering: The system serves the exact same high-poly GLB to a flagship iPhone 16 Pro as it does to a 4-year-old budget Android phone.
Conversion Latency: If GLB-to-USDZ conversion is executed synchronously on-demand during a user's first iOS click, it will result in a 10-to-20 second loading lag, causing immediate user drop-off.
13. Current Maturity Assessment
Early MVP-Level / Sandbox Stage. The core 3D viewer is stable, clean, and production-ready. However, the AR visualization architecture is highly fragile, platform-siloed, and completely lacks real-world physical scaling logic. It cannot be safely released to arbitrary creators or general consumers in its current state.

14. What Is Already Production-Ready
The non-mutating scaling and grounding calculations in ProductModel.jsx.
The unified lighting, control boundaries, and shadow management in ProductCanvas.jsx.
The core table index structures in schema.sql.
15. What Is Fragile
ARViewer.jsx: The raw, custom hit-testing canvas is prone to browser compatibility crashes and fails entirely on iOS.
Local Blob uploads: The mock uploads in the form will break when shifted to production network conditions.
16. What Should Be Preserved
The mathematical grounding logic that maps model minimum Y bounds to y = 0.
The modular structure of the React form pages and dashboard layout.
The centralized database indexes.
17. What Must Be Rewritten
The AR Launch Architecture: The raw, custom @react-three/xr R3F setup should be replaced with a highly optimized, industry-standard <model-viewer> approach for the user-facing AR launch gateway.
Model Scale Logic for AR: The system must decouple framing scale (used to make the model look good in a desktop/mobile 3D box) from physical scale (used to render the model at exactly 1:1 real-world dimensions in AR).
18. What Should NOT Be Overengineered
No Custom WebXR SLAM Engine: Do not write custom surface trackers, lighting estimators, or room scanners. Leverage pre-existing browser/OS native layers.
No Real-Time Cloud Conversion: Keep the backend conversions simple, file-based, and asynchronous.
19. What Is Missing Entirely
iOS Quick Look Integration: A .usdz asset delivery path for native iOS AR.
Backend GLB-to-USDZ Converter Pipeline: An automated conversion service (e.g., using usd-from-gltf or gltf2usd packages) triggered upon product upload.
Physical Scale Registry: Fields in the creator upload form allowing them to input the product's actual height, width, or depth in centimeters/meters.
AR Session Telemetry: Logging for AR session durations, placement success rates, and device hardware/browser environment characteristics.
20. Recommended Architecture Direction
To scale ScanVista to a reliable, consumer-grade platform, we recommend migrating to a Hybrid <model-viewer> & Native Quick Look Architecture:

[ User Requests AR ]
                                  │
                  Is the device iOS or Android/WebXR?
                     /                         \
             [ iOS Safari ]               [ Android Chrome ]
                  │                                 │
         Launch Apple AR Quick Look        Launch Google WebXR
         Using native .usdz asset       Using `<model-viewer>` GLB
Recommended Frontend Implementation:
Leverage <model-viewer>: Use Google's standardized web component strictly for the AR gatekeeper. It natively bridges both Android's WebXR Scene Viewer and iOS's Quick Look (via the ios-src property) with maximum reliability, touch control, lighting estimation, and automatic fallback.
Decouple Viewer and AR Scale:
The R3F 3D Canvas continues to use the non-mutating scaleFactor to beautifully frame the model within the screen box.
The AR Button serves the raw, unmodified GLB/USDZ file (with the scale property matching actual real-world measurements) directly to the native device viewer, guaranteeing 1:1 real-world scale placement.
Recommended Backend Implementation:
Extend Database Schema:
Add usdz_url (TEXT) to the products table.
Add real_height_meters (NUMERIC) to standardise model scaling.
Implement Asynchronous USDZ Pipeline:
When a creator uploads a GLB, the server processes it:
Compresses the GLB (Draco compression + resizing textures to max 2K for mobile safety).
Asynchronously converts the GLB to USDZ on the backend.
Saves both optimized assets to Supabase Storage and updates the database columns.
